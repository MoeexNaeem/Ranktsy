Summary

Rankkw is a keyword-research and SEO analytics tool for online sellers. It uses the Google Ads API for exactly one purpose: to read historical keyword search volume from Keyword Planner and show it to a seller researching what to make and how to title it. It does not create, edit, or manage advertising campaigns of any kind.

Why this is a low-risk integration

The entire integration is a single read-only call — generateKeywordHistoricalMetrics — against the tool operator's own account. No mutate operations, no campaign or budget access, no management of third-party ad accounts, and no write access of any kind. The scope requested is the minimum the feature needs.

What the tool does

Sellers on marketplaces such as Etsy get no search-volume data from those marketplaces. Rankkw fills that gap: a seller types a product keyword (e.g. “ceramic coffee mug”) and sees how many people actually search for it each month, how that demand moves across the year, and roughly where the searchers are. That helps them decide which products and keywords are worth their time before they invest in making and listing an item.

All other data in Rankkw comes from the seller's marketplace API. The Google Ads API contributes precisely three things, each clearly labelled in the interface as sourced from Google:

  • Average monthly search volume for a keyword.
  • A 12-month trend line of that volume, to reveal seasonality.
  • A rough country split for a keyword, across a small fixed set of markets.

Users

The users are independent online sellers and small shops doing product and keyword research. They authenticate to Rankkw with their own Rankkw account. They do not connect a Google Ads account and never see the Google Ads API directly — the keyword-volume lookups run server-side against the tool operator's single developer account. From the user's side, Google data simply appears as an extra column and chart alongside their marketplace research.

API services and methods called

The integration uses one service and one method. Nothing else in the Google Ads API is called.

API call design

Rankkw calls the REST interface directly (no client library). A short-lived OAuth access token is minted from a stored refresh token and cached in memory for its lifetime (~55 minutes) to avoid re-minting on every request. The request body targets one geo and the Google Search network:

POST https://googleads.googleapis.com/v20/customers/{customerId}:generateKeywordHistoricalMetrics # headers Authorization: Bearer {short-lived access token} developer-token: {developer token} login-customer-id: {manager (MCC) id, when the query account sits under one} # body { "keywords": ["ceramic coffee mug", "handmade mug", ...], # up to 1000 "geoTargetConstants": ["geoTargetConstants/2840"], # e.g. US "keywordPlanNetwork": "GOOGLE_SEARCH", "language": "languageConstants/1000" # English }

From the response, only three fields per keyword are read and displayed: avgMonthlySearches, competition, and monthlySearchVolumes (the 12-month series). The geo set is a small fixed list — United States, United Kingdom, Canada, Australia, Germany, France — used both for single-geo lookups and, for the country split, one lookup per market that is then normalised to percentages.

Where the data appears (Required Minimum Functionality)

Google keyword data is shown inside a genuine, functional product — not re-exported raw. It surfaces in three places, each labelled as Google-sourced:

SurfaceWhat it shows
Keyword ToolA “Google Searches” column beside each researched keyword — the real average monthly volume.
Monthly TrendsThe 12-month search-volume curve, with the peak month highlighted so sellers list ahead of demand.
Searchers by CountryThe normalised country split for a keyword across the six supported markets.

Around this Google data, Rankkw provides substantial independent functionality — keyword difficulty, competition analysis, listing audits, tag research, shop analytics — so the Google Ads data enhances a full tool rather than being the whole product.

Data handling & compliance

  • No long-term storage of Google data. Keyword-metric responses are held only in a short-lived server cache to serve a request quickly, then discarded. Google data is never resold, exported in bulk, or offered as a standalone data feed.
  • Always attributed. Every figure derived from the Google Ads API is labelled in the UI as sourced from Google, and is never presented as marketplace data.
  • No blending that misrepresents source. Google search volume and marketplace metrics are shown side by side but kept visually distinct and separately labelled.
  • Graceful, honest degradation. When the API is unavailable or unconfigured, the relevant fields show “—” rather than an estimated or fabricated number. Rankkw’s guiding rule is that no displayed number is invented.
  • Credentials are server-side only and never exposed to the browser. Users do not supply Google credentials.
  • Version resilience. The API version is configurable; the tool detects a sunset version (HTTP 404) and surfaces an actionable error rather than failing silently.
Request volume

Lookups are triggered by an individual seller's on-demand keyword research and are cached, so call volume scales with active research sessions, not with a crawl. Basic access limits comfortably cover this usage. We do not batch-harvest the keyword corpus.

Access level requested, and why

Basic access. The tool needs real (non-test) Keyword Planner volume to be useful — test accounts return no live data. Usage is read-only, on-demand, cached, and limited to a single method, which fits within Basic access limits. We are not requesting Standard access: we do not manage client ad accounts and have no need for the higher operation volumes that tier is intended for.
